# @cosyte/ccda — Roadmap (v1)

North star: **A developer can parse a real-world, vendor-quirky C-CDA document and pull useful sections out of it in one line — without having read the C-CDA Implementation Guide.**

- **Granularity:** standard (8 phases, 3–5 plans each anticipated)
- **Mode:** yolo (auto-advance enabled)
- **Parallelization:** enabled — plans within a phase may run in parallel where they touch disjoint modules
- **Coverage:** 130/130 v1 REQ-IDs mapped to exactly one phase (MODEL-05 deferred to v2 and not counted)

> **Note on REQ-ID prefixes.** REQUIREMENTS.md uses the `DOC-` prefix for two distinct categories: the *Document Type & Header* group (`DOC-01..06`) and the *Documentation* group (`DOC-01..18`). Throughout this roadmap, header IDs are referenced as `DOC-01h..DOC-06h` and documentation IDs as `DOC-01d..DOC-18d`. The Traceability table in REQUIREMENTS.md uses the same disambiguation.

> **Revision note (2026-04-22, post-research).** Phase 3 gains CODE-01 + CODE-02 (OID registry lands with TYPES-02 so `codeSystemName` resolves from day one). Phase 4 narrows to templates only. Phase 7 gains CODE-03 + two new HIE profiles (BIP-07 carequality, BIP-08 commonwell). New security REQs land in Phase 2 (PARSE-09, TOL-07, TOL-08) and Phase 8 (TEST-10, DOC-17d, DOC-18d). USCDI v3 helpers + filter helpers + `doc.summary()` land in Phase 5 (HELPERS-11..17).

## Phases

- [ ] **Phase 1: Project Foundation & XML Parser ADR** — Scaffold the repo (build, lint, test, dual ESM+CJS, strict TS) AND lock the XML-parser runtime-dep choice via ADR before any parser code is written. Research has pre-surfaced `@xmldom/xmldom ^0.9.10` as the likely ADR outcome; the discuss step ratifies or diverges.
- [ ] **Phase 2: Core XML Parser & Tolerance** — `parseCCDA(raw)` decodes well-formed C-CDA R2.1 XML (namespaces, mixed content, IDREF, base64, BOM, processing instructions) with lenient default + strict mode + warnings/errors registry; ships security-by-default (DTD rejection, billion-laughs defense, PHI-safe warning snippets, decode-size bounds).
- [ ] **Phase 3: Document Header, Typed Model, Types & OIDs** — `doc.type` / `doc.patient` / header participants + immutable typed `CCDADocument` model (structural-sharing mutation contract) + RIM-derived composite data types (CD, II, AD, PN, TS, PQ, ED, etc.) carrying `nullFlavor` + built-in OID registry so `CodedValue.codeSystemName` resolves from day one.
- [ ] **Phase 4: Templates & Section Resolution** — Built-in template registry (`defineTemplate()`) authored for every enumerated C-CDA R2.1 section and entry template; template-first / LOINC-fallback section access; registry composition with override semantics.
- [ ] **Phase 5: Named Helpers & Narrative Reconciliation** — One-line DX (`doc.problems.active`, `doc.medications.current`, `doc.allergies`, etc.) + USCDI v3 helpers (`doc.familyHistory`, `doc.advanceDirectives`, `doc.functionalStatus`, `doc.healthConcerns`, `doc.goals`) + filter helpers (`byCode`, `byRoute`, `activeAt`, `byCategory`, `latest`, `byDateRange`) + `doc.summary()` rollup + narrative-content tree + entry↔narrative IDREF reconciliation (lazy on access) with mismatch warnings.
- [ ] **Phase 6: Serialization & Round-Trip** — `doc.toString()`, `doc.toJSON()`, `doc.prettyPrint()`, `buildDocument(...)` (CCD-only in v1) produce namespace-clean canonical C-CDA R2.1 XML and preserve semantics across parse → mutate → serialize → parse.
- [ ] **Phase 7: Profile System & Built-ins** — `defineProfile()` API with merge/extend semantics + 7 built-in profiles (epic, cerner, meditech, athena, generic, carequality, commonwell) + profile-scoped OID-registry extension.
- [ ] **Phase 8: Testing Hardening, Examples, Starter Kit & Documentation** — Canonical/edge-case/vendor-quirk/security/profile-authoring/template-authoring tests at ≥ 90% line coverage, 3 runnable examples, publishable profile starter kit, SECURITY.md, and the complete README + ancillary docs.

## Phase Details

### Phase 1: Project Foundation & XML Parser ADR
**Goal**: A developer cloning the repo can install, build, typecheck, lint, and test with a single command sequence; downstream phases never have to revisit tooling AND never have to re-litigate the XML-parser choice — the ADR is on disk before any parser code is written.
**Depends on**: Nothing (first phase)
**Requirements**: SETUP-01, SETUP-02, SETUP-03, SETUP-04, SETUP-05, SETUP-06, DOC-15d, DOC-16d
**Success Criteria** (what must be TRUE):
  1. A developer can run `pnpm install && pnpm build && pnpm typecheck && pnpm lint && pnpm test` from a clean clone and every command exits 0 with zero warnings.
  2. A developer importing the package from an ESM project and another from a CJS project both resolve the correct entry through the `exports` map and receive typed IntelliSense.
  3. A developer inspecting `package.json` sees `"type": "module"`, dual-build artifacts declared, Node 18+ engines field, and every entry under `dependencies` justified by an ADR under `.planning/adr/` (total runtime deps ≤ 3, pinned version floors above all known CVEs).
  4. A developer opens `.planning/adr/0001-xml-parser.md` and sees a committed ADR naming the chosen XML parser (research pre-recommended `@xmldom/xmldom ^0.9.10`; candidates considered: `fast-xml-parser`, `sax`, `@xmldom/xmldom`, `libxmljs2`), alternatives considered, licensing and maintenance signals each cleared, the version floor, and the bar — namespace-aware (`(namespaceURI, localName)` tuple access), mixed-content preservation with order, DTD processing disabled by default, thin typed-accessor adapter contract in `src/xml/adapter.ts` — and zero parser source files exist outside of stubs.
  5. A developer editing any `.ts` file gets strict-mode errors for `any`, unchecked index access, and missing types from their editor immediately; LICENSE (MIT) is at repo root.
**Plans**: 4 plans
**UI hint**: no

### Phase 2: Core XML Parser & Tolerance
**Goal**: A developer calling `parseCCDA(raw)` on any well-formed C-CDA R2.1 XML document — including vendor-quirky input — receives a structurally correct parse result with stable, positional warnings surfaced for every known deviation, and structurally corrupt input throws typed fatal errors with line/column/snippet context. Security-by-default: DTD processing disabled, billion-laughs bounded, decode sizes capped, warning snippets PHI-safe.
**Depends on**: Phase 1
**Requirements**: PARSE-01, PARSE-02, PARSE-03, PARSE-04, PARSE-05, PARSE-06, PARSE-07, PARSE-08, PARSE-09, TOL-01, TOL-02, TOL-03, TOL-04, TOL-05, TOL-06, TOL-07, TOL-08
**Success Criteria** (what must be TRUE):
  1. A developer can parse a well-formed C-CDA R2.1 document supplied as a string or Buffer (UTF-8, with or without BOM) and receive a `CCDADocument` whose XML namespaces (default `urn:hl7-org:v3`, `voc`, `sdtc`, `xsi`) are correctly resolved regardless of vendor prefix variation; downstream accessors address elements by `(namespaceURI, localName)` tuples, never prefix strings.
  2. A developer parsing a document with mixed-content narrative (ordered children preserved), `<?xml-stylesheet?>` processing instructions, intra-document `ID`/`IDREF` references (resolved via a dedicated ID index, not `getElementById`), or base64-encoded `<observationMedia>` attachments sees every child node preserved, processing instructions retained for round-trip, IDREF resolutions exposed, and base64 payloads exposed lazily as raw bytes with `{ maxDecodedBytes }` cap enforcement.
  3. A developer parsing a document with a known Tier 2 deviation (missing templateId, unknown templateId, narrative-only section, unresolved IDREF, OID not in registry, mixed-content deviation, namespace prefix variation, embedded HTML in narrative, base64 attachment, missing `xsi:type` on observation value, `xsi:type` resolving to a non-HL7 namespace, etc.) gets a parsed document in lenient mode plus `doc.warnings` entries with stable codes and XPath-ish positional context — and receives `onWarning` callbacks as they are emitted.
  4. A developer parsing a structurally corrupt document (not XML, no `<ClinicalDocument>` root, invalid namespace, empty input) receives a thrown `CCDAParseError` with a stable code (`NOT_XML`, `NO_CLINICAL_DOCUMENT_ROOT`, `INVALID_NAMESPACE`, `EMPTY_INPUT`), `position` (line/column), and `snippet` (PHI-redacted by default) — even in lenient mode. A document with a `<!DOCTYPE>` declaration has the DTD discarded, `CCDA_DTD_REJECTED` emitted, no external entity resolution attempted. A document triggering entity expansion beyond `maxEntityCount` emits `CCDA_ENTITY_EXPANSION_EXCEEDED`; a document decoding past `maxDecodedBytes` emits `CCDA_DECODE_SIZE_EXCEEDED`.
  5. A developer opting into `parseCCDA(raw, { strict: true })` gets every Tier 2 deviation aggregated and thrown as a typed `CCDAValidationError` carrying all violation codes and positional contexts. A developer piping warnings to logs with default options sees snippets that strip text-node content — name-and-attribute context preserved, PHI redacted to `<…>` — unless they explicitly set `{ redactSnippets: false }`.
**Plans**: 5 plans
**UI hint**: no

### Phase 3: Document Header, Typed Model, Types & OIDs
**Goal**: A developer accessing a parsed `CCDADocument` can navigate the header (`doc.type`, `doc.patient`, `doc.author[]`, `doc.encounter`, etc.) and the body (`doc.sections`, `doc.section(templateId|loinc)`, entries) by typed accessors, receive parsed RIM composite data types (CD, II, AD, PN, TS, PQ, ED, etc.) carrying `nullFlavor` alongside value, resolve `CodedValue.codeSystemName` through the built-in OID registry on every read, and mutate the document only via explicit methods returning a new instance (structural sharing).
**Depends on**: Phase 2
**Requirements**: DOC-01h, DOC-02h, DOC-03h, DOC-04h, DOC-05h, DOC-06h, MODEL-01, MODEL-02, MODEL-03, MODEL-04, MODEL-06, MODEL-07, TYPES-01, TYPES-02, TYPES-03, TYPES-04, TYPES-05, TYPES-06, CODE-01, CODE-02
**Success Criteria** (what must be TRUE):
  1. A developer can read `doc.type` (one of the 12 C-CDA document types or `'unknown'`), `doc.templateIds`, `doc.effectiveTime`, `doc.languageCode`, `doc.title`, `doc.confidentialityCode`, `doc.setId`, and `doc.versionNumber` on any parsed document; absent fields return `undefined` and never throw. `diagnostic-imaging` and `unstructured-document` are detected but their helper surface is intentionally thin.
  2. A developer can read `doc.patient.mrn` (with vendor-OID-aware picking), `doc.patient.fullName`, `doc.patient.dateOfBirth` (parsed `Date`), `doc.patient.address`, and the rest of the patient contract — plus `doc.author[]`, `doc.custodian`, `doc.informant[]`, `doc.legalAuthenticator`, `doc.authenticator[]`, `doc.dataEnterer`, `doc.informationRecipient[]`, and `doc.encounter` (nullable) — on any document with the corresponding header participants; absent participants return `undefined` or `[]`.
  3. A developer can call `doc.sections` to iterate every top-level section in document order, `doc.section(templateId)` for primary template lookup (root or root+extension), `doc.section(loincCode)` for LOINC-fallback lookup, and `section.rawXml` as the escape hatch for values the typed helpers don't cover (the former "xpath-ish" `doc.get(path)` is deferred to v2). Template match always wins over code match.
  4. A developer importing the library receives typed interfaces for every C-CDA / RIM composite type used by the library (`CD`, `CE`, `CWE`, `II`, `AD`, `PN`, `TEL`, `TS`, `IVL_TS`, `EIVL_TS`, `PQ`, `ED`, `ST`, `BL`, `INT`, `REAL`) — each carrying a `nullFlavor?: NullFlavor` field. `CodedValue` exposes `code`, `codeSystem`, `codeSystemName` (resolved from day one via the built-in OID registry: SNOMED, LOINC, RxNorm, ICD-10-CM/PCS, CPT, CVX, UCUM, NDC, HL7 Administrative Gender / Marital Status / Race / Ethnicity), `displayName`, `originalText`, `translations[]`, `nullFlavor?`. Unparseable timestamps and quantities return `undefined` for the typed getter (no throw); raw string remains accessible.
  5. A developer mutating a document via `setHeaderField`, `addSection`, `removeSection`, `addEntry`, `removeEntry`, or `setEntryField` receives a **new** `CCDADocument` instance (structural sharing; `doc1 !== doc2`); the original instance and any retained references to it remain unchanged. Round-trip semantics: `doc2.toString()` reflects the mutation; `doc1.toString()` does not.
**Plans**: 5 plans
**UI hint**: no

### Phase 4: Templates & Section Resolution
**Goal**: A developer can rely on a built-in template registry covering every enumerated C-CDA R2.1 section template (Core + Discharge Summary + Care Plan + Operative Note + Diagnostic Imaging groups) and entry template (Problem/Medication/Allergy/Result/Vital/Immunization/Encounter/Procedure/Social History/Care Plan/Family History/Functional Status/Plan of Treatment families), all authored through the public `defineTemplate()` API, with lookup by `templateId` (root + optional `@extension`) primary and LOINC code fallback.
**Depends on**: Phase 3
**Requirements**: TPL-01, TPL-02, TPL-03, TPL-04, TPL-05
**Success Criteria** (what must be TRUE):
  1. A developer importing the library receives every C-CDA R2.1 built-in section template enumerated in REQ TPL-01 — including Core templates (Problems, Medications, Allergies, Immunizations, Results, Vital Signs, Encounters, Procedures, Social History, Plan of Treatment, Assessment/Plan, Reason for Referral, Reason for Visit, Instructions, Medical Equipment, Functional Status, Mental Status, Advance Directives, Payers, Chief Complaint, HPI, Past Medical History, Family History, Review of Systems, Physical Exam, General Status, Nutrition), the Discharge Summary family (Hospital Course, Hospital Discharge Diagnosis, Discharge Medications entries-required, Hospital Discharge Instructions, Hospital Discharge Studies Summary, Hospital Admission Diagnosis, Admission Medications, Hospital Consultations, Hospital Discharge Physical), the Care Plan family (Health Concerns, Goals, Interventions V3, Health Status Evaluations and Outcomes), the Operative Note family (Anesthesia, Complications, Pre/Postop Diagnosis, Procedure Description/Findings/Specimens/Disposition/Indications, Estimated Blood Loss, Surgical Drains, Planned Procedure, Procedure Implants), and the Diagnostic Imaging family (Findings, DICOM Object Catalog).
  2. A developer importing the library receives every C-CDA R2.1 built-in entry template enumerated in REQ TPL-02 — including Problem Concern Act V3 + Problem Observation V3, Medication Activity V2 + Medication Information V2 + Medication Supply Order V3 + Indication V2, Allergy Concern Act V3 + Allergy–Intolerance Observation V2 + Reaction Observation V2 + Severity Observation V2, Result Organizer V3 + Result Observation V3, Vital Signs Organizer V2 + Vital Sign Observation V2, Immunization Activity V3 + Immunization Medication Information V2, Encounter Activity V3 + Encounter Diagnosis V3 + Service Delivery Location, Procedure Activity Act/Observation/Procedure V2, Social History Observation V3 + Smoking Status V2 + Tobacco Use V2 + Pregnancy Observation, Care Plan entries (Goal Observation, Intervention Act V2, Health Concern Act V2, Outcome Observation V2), Family History Observation/Organizer V3, Functional Status Observation V2 + Cognitive Status Observation V2, Plan of Treatment entries (Planned Observation V2, Planned Procedure V2, Planned Medication Activity V2).
  3. A developer calling `defineTemplate({ name, templateId, extension?, parse, entryShape? })` registers a new template; duplicate registration throws `TemplateDefinitionError` unless `{ override: true }` is set, and developer-registered templates take precedence over built-ins on override.
  4. A developer accessing a section sees template lookup performed against `templateId` (root + optional `@extension`) as a pure registry lookup keyed by `\`${root}|${extension ?? ""}\``; LOINC code is a fallback only and never wins over a template match. Entry wrappers expose a discriminated union keyed by `kind` string; unrecognized entries parse into a bounded `{ kind: 'unknown', xsiType, templateIds, raw }` fallback shape.
  5. A developer reviewing the codebase can `grep "defineTemplate("` and find both built-ins and user-registered templates through the same API surface — the built-ins are authored exactly the way a profile starter kit author would author their own.
**Plans**: 3 plans
**UI hint**: no

### Phase 5: Named Helpers & Narrative Reconciliation
**Goal**: A developer can fulfill the north star — one-line extraction of common C-CDA fields — through `doc.problems.active`, `doc.medications.current`, `doc.allergies`, `doc.immunizations`, `doc.results.recent`, `doc.vitals.latest`, `doc.encounters`, `doc.procedures`, `doc.socialHistory.smoking`, plus USCDI v3 helpers (`doc.familyHistory`, `doc.advanceDirectives`, `doc.functionalStatus`, `doc.healthConcerns`, `doc.goals`), filter helpers (`byCode`, `byRoute`, `activeAt`, `byCategory`, `latest`, `byDateRange`), and the single-call `doc.summary()` dashboard rollup — and can walk the narrative content tree, reconcile entry IDREFs against narrative `content[@ID]` spans lazily on access, and surface mismatches as `CCDA_NARRATIVE_ENTRY_MISMATCH` warnings scoped to three specific entry-type categories without ever silently rewriting narrative.
**Depends on**: Phase 4
**Requirements**: HELPERS-01, HELPERS-02, HELPERS-03, HELPERS-04, HELPERS-05, HELPERS-06, HELPERS-07, HELPERS-08, HELPERS-09, HELPERS-10, HELPERS-11, HELPERS-12, HELPERS-13, HELPERS-14, HELPERS-15, HELPERS-16, HELPERS-17, NARR-01, NARR-02, NARR-03, NARR-04
**Success Criteria** (what must be TRUE):
  1. A developer can read `doc.problems.active` (SNOMED-status-matched, not narrative-scraped), `doc.medications.current` (half-bounded intervals tolerated), `doc.allergies` (with `isNoKnownAllergies` flag that distinguishes "absent section" from "explicitly-stated no known allergies"), `doc.immunizations`, `doc.results.recent`, `doc.vitals.latest` (conventional panel; missing slots `undefined`, never fabricated), `doc.vitals.byLoinc(code)`, `doc.encounters`, `doc.procedures`, `doc.socialHistory.smoking`, and `doc.socialHistory.byLoinc(code)` — each in one line, returning typed arrays / objects with the documented field shapes (CodedValue, parsed dates, units, status, performer/prescriber, etc.).
  2. A developer can read USCDI v3 helpers on any document with the corresponding sections: `doc.familyHistory` (Family History Section), `doc.advanceDirectives` (Advance Directives Section V3), `doc.functionalStatus` (Functional Status Section V2), `doc.healthConcerns` (Health Concerns Section V2 — Care Plan), `doc.goals` (Goals Section — Care Plan). Each returns typed arrays; absent sections return `[]` and never throw.
  3. A developer can filter across the collection helpers in a single line: `doc.problems.byCode(...)`, `doc.medications.byRoute(...)`, `doc.medications.activeAt(date)`, `doc.results.byCategory(...)`, `doc.results.byCode(...)`, `doc.results.latest(loincCode)`, `doc.encounters.byDateRange(from, to)`. A developer can also call `doc.summary()` once and receive a full dashboard rollup (`patient, activeProblems, currentMedications, allergies, lastVitals, recentResults, lastEncounter`) without writing any filter code.
  4. A developer accessing `section.text` sees the narrative as a structured content tree (paragraphs, tables, lists, `<content ID="...">` spans, inline HTML-like elements) — not a flat string — and that narrative is byte-for-byte preserved on serialize (the library never rewrites narrative content from structured entries). `section.isNarrativeOnly` is queryable as a boolean on every section.
  5. A developer reading `section.matchedEntries`, `section.unmatchedEntries`, or `section.orphanNarrative` triggers lazy reconciliation (no user `reconcile()` call required); `CCDA_UNRESOLVED_IDREF` emits for entries whose `text/reference[@value]` does not match any narrative `content[@ID]`; `CCDA_NARRATIVE_ENTRY_MISMATCH` emits only for medication dose mismatches, problem code-displayName vs narrative-text mismatches (where narrative-text is non-empty), and result numeric-value mismatches.
**Plans**: 5 plans
**UI hint**: no

### Phase 6: Serialization & Round-Trip
**Goal**: A developer can take a parsed, mutated, or constructed `CCDADocument` and emit namespace-clean canonical C-CDA R2.1 XML — or a JSON / pretty-printed summary — such that parse → modify → serialize → parse yields a structurally equivalent document. Outbound construction (`buildDocument`) is scoped to CCD documents in v1.
**Depends on**: Phase 3, Phase 5
**Requirements**: SER-01, SER-02, SER-03, SER-04, SER-05
**Success Criteria** (what must be TRUE):
  1. A developer calling `doc.toString()` on any parsed document (including vendor-quirky input) receives namespace-clean, canonical C-CDA R2.1 XML with default namespace forced to `urn:hl7-org:v3`, vendor prefix aliases stripped, stylesheet processing instructions preserved, and no leaked mixed-content quirks.
  2. A developer running `parseCCDA(doc.toString())` on every canonical fixture (all 12 document types) receives a `CCDADocument` structurally equivalent to the original — verified by a dedicated structural-equivalence walker over the typed model (header + sections + entries + narrative tree), not byte-diff.
  3. A developer calling `doc.toJSON()` receives a structured JSON representation of the full document (header + sections + entries) suitable for snapshotting or cross-process transport, and `doc.prettyPrint()` returns a tight human-readable multi-line summary: header line + per-section `"Name (templateId): N entries"` line + helper-rollup line.
  4. A developer using `buildDocument({ type: 'ccd', patient, ... }).addSection(...).addEntry(...).toString()` constructs a valid outbound **CCD** document for tests and small tools; non-CCD outbound document types throw `TypeError` (deferred to v2).
  5. A developer round-tripping a document parsed under any vendor profile and re-serializing receives namespace-clean C-CDA R2.1 — profile quirks affect parsing, never serialization.
**Plans**: 4 plans
**UI hint**: no

### Phase 7: Profile System & Built-ins
**Goal**: A developer can define, extend, and compose vendor/integration profiles via a first-class public API, apply them to parses, and rely on 7 ready-made profiles (epic, cerner, meditech, athena, generic, carequality, commonwell) that reduce warnings against realistic vendor / HIE shapes — and the OID-registry / template-registry extensions registered through profiles compose with the built-ins from Phase 3 and Phase 4.
**Depends on**: Phase 4, Phase 6
**Requirements**: PROF-01, PROF-02, PROF-03, PROF-04, PROF-05, PROF-06, PROF-07, PROF-08, PROF-09, BIP-01, BIP-02, BIP-03, BIP-04, BIP-05, BIP-06, BIP-07, BIP-08, CODE-03
**Success Criteria** (what must be TRUE):
  1. A developer calling `defineProfile({ name, ... })` with valid input receives a readonly `Profile` exposing `name`, `description`, `oidRegistry`, `customTemplates`, `lineage`, and `describe()`; invalid input (bad option shapes, duplicate custom OIDs, unknown option keys) throws `ProfileDefinitionError` with an actionable message.
  2. A developer using `extends: parentProfile` or `extends: [p1, p2]` receives a profile whose merged options follow documented semantics: leftmost-ancestral → rightmost-layered → `defineProfile` body final word. Scalars overwrite; arrays concat+dedupe; `oidRegistry` / `customTemplates` deep-merge per key; `onWarning` handlers chain in leftmost-first order. Type signature: `Profile | readonly Profile[]`, never `any[]`.
  3. A developer calling `parseCCDA(raw, profile)` sees `doc.profile?.name` and `doc.profile?.lineage` populated, profile-registered OIDs resolved through `CodedValue.codeSystemName`, and profile-registered custom templates extending the template registry for that parse only. Profile state flows via parser context, not module globals (explicit-argument calls never touch any default profile state).
  4. A developer calling `setDefaultProfile(p)` / `getDefaultProfile()` / `setDefaultProfile(null)` can manage a process-scoped default; explicit arguments override and `parseCCDA(raw, { profile: null })` opts out for a single call.
  5. A developer importing `profiles.epic`, `profiles.cerner`, `profiles.meditech`, `profiles.athena`, `profiles.generic`, `profiles.carequality`, or `profiles.commonwell` and parsing a realistic vendor-shape (or HIE-shape) fixture with the profile receives fewer warnings than parsing the same fixture without a profile; each built-in is authored through the public `defineProfile()` API.
**Plans**: 5 plans
**UI hint**: no

## Parallelization Notes

Within each phase, plans that touch disjoint modules may run in parallel; plans that share a module must serialize. Concrete expectations:

- **Phase 1:** Toolchain plans (tsup config, Vitest config, ESLint+Prettier, tsconfig + strict flags, package.json exports + scripts, README skeleton, LICENSE) are largely independent and can run in parallel. **The XML-parser ADR is a discuss-phase deliverable that gates everything in Phase 2 — no parser source code may be written before `0001-xml-parser.md` is committed.** Research has pre-surfaced `@xmldom/xmldom ^0.9.10` as the recommended outcome; discuss-phase ratifies or diverges. A final smoke-test plan runs last to verify the full `install/build/typecheck/lint/test` pipeline.
- **Phase 2:** The chosen XML parser is wired into the adapter boundary in `src/xml/adapter.ts` first (all other directories import only from `src/xml/index.ts`, so the library can be swapped later). Namespace handling, processing-instruction preservation, mixed-content handling, IDREF indexing (not `getElementById` — C-CDA ships no DTD), base64 lazy decode with size caps, BOM/charset normalization, DTD rejection, billion-laughs defense, and PHI-safe snippet redaction can then proceed in parallel against shared fixtures. The warnings/error-code registry and `onWarning` plumbing should be built early and consumed by each parser plan; strict-mode escalation is a capstone plan.
- **Phase 3:** Composite type parsers (CD, CE, CWE, II, AD, PN, TEL, TS/IVL_TS, PQ, ED) are independent and parallelizable; each carries `nullFlavor` first-class. The built-in OID registry is one plan and lands in Phase 3 (not Phase 4) so `CodedValue.codeSystemName` resolves from day one. The `CCDADocument` shell, header parsers (patient with vendor-OID-aware MRN picking, author, custodian, encounter), and section/entry wrapper traversal are serial dependencies. Mutation methods (`setHeaderField`, `addSection`, `removeSection`, `addEntry`, `removeEntry`, `setEntryField`) are a final plan; the structural-sharing mutation contract is pre-locked (research recommendation) but ratified at a discuss step before mutation code is written.
- **Phase 4:** `defineTemplate()` API + duplicate-registration throw + override + the `Map<\`${root}|${extension ?? ""}\`, Template>` two-layer registry is the foundational plan. Built-in section-template definitions (Core + Discharge Summary + Care Plan + Operative Note + Diagnostic Imaging families) parallelize as independent files; same for entry templates (Problem/Medication/Allergy/Result/Vital/Immunization/Encounter/Procedure/Social History/Care Plan/Family History/Functional Status/Plan of Treatment families).
- **Phase 5:** Helper builders (`problems`, `medications`, `allergies`, `immunizations`, `results`, `vitals`, `encounters`, `procedures`, `socialHistory`, `familyHistory`, `advanceDirectives`, `functionalStatus`, `healthConcerns`, `goals`) are mutually independent and parallelizable. Filter helpers (`byCode`, `byRoute`, `activeAt`, `byCategory`, `latest`, `byDateRange`) layer on top of the collection helpers. `doc.summary()` is a capstone plan that composes them. The narrative-content tree builder + lazy IDREF reconciler + `isNarrativeOnly` boolean are a serial trio; the `CCDA_NARRATIVE_ENTRY_MISMATCH` emit-site (scoped to three entry-type categories) lands once reconciliation is in place.
- **Phase 6:** `toString()` and `toJSON()` can run in parallel (disjoint emitters). `prettyPrint()` and the CCD-only `buildDocument()` are independent. The round-trip fixture sweep (structural-equivalence walker) is a final plan.
- **Phase 7:** `defineProfile()` core + validation errors is the first plan; `extends`/merge semantics and default-profile management can then parallelize. The seven built-in profiles (epic, cerner, meditech, athena, generic, carequality, commonwell) are mutually independent and all parallelizable once the API surface stabilizes; the per-vendor warning-reduction sweep is a capstone. CODE-03 (profile-scoped OID-registry extension) lands here as a small plumbing plan.
- **Phase 8:** Fixture authoring (canonical, edge-case, vendor-quirk, security, profile-authoring, template-authoring) parallelizes across categories. The three examples are independent. Starter-kit assembly is one plan; README authoring decomposes into quickstart + feature list, access patterns, cookbook, templates+profiles section, tolerance section, error handling, security section, contributing/footer — most of which parallelize. `SECURITY.md` is one plan. CHANGELOG and any per-runtime-dep ADRs beyond the Phase 1 XML-parser ADR are trivially parallel.
